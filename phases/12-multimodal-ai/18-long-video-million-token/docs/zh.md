# 百万 Token 上下文的长视频理解

> 一个 1 小时 4K 视频在 24 FPS 下，做 patch 和嵌入，产生约 6000 万个 token。转写一个 2 小时播客是 30000 个 token。一部完整的蓝光电影，即使做激进池化压缩后，也是数十万个 token。Google 的 Gemini 1.5（2024年3月）以 1000 万 token 上下文开启了这个时代，在小时级视频上做可靠的草堆找针回忆。LWM（Liu 等人，2024年2月）展示了环形注意力的扩展路径。LongVILA 和 Video-XL 进一步扩展了摄入。VideoAgent 用代理式检索替换了原始上下文。每种方法在算力、回忆和工程复杂性上做出不同的权衡。本课将它们并列阅读。

**Type:** Build
**Languages:** Python (stdlib, needle-in-haystack simulator + agentic-retrieval router)
**Prerequisites:** Phase 12 · 17 (video temporal tokens)
**Time:** ~180 minutes

## 学习目标

- 计算不同 FPS 和池化下长格式视频的总视觉 token 数。
- 解释三种扩展路径：暴力上下文（Gemini 1.5）、环形注意力（LWM）、token 压缩（LongVILA / Video-XL）。
- 在准确率和延迟上比较原始上下文视频 VLM 和代理式检索视频 VLM（VideoAgent）。
- 为 30 分钟视频设计草堆找针测试，并测量在特定分钟的回忆率。

## 问题

Qwen2.5-VL 规模大小的单帧在原生的 384 下约 729 个 token。3x3 池化后每帧 81 个 token。30 分钟片段在 1 FPS = 1800 帧 = 145,800 个 token。2025 年开源 VLM 可行，但很紧。在 2 FPS 下，291,600 个 token——只有最大上下文能容纳。

2 小时电影在 1 FPS 下是 583k 个 token。超出大多数 2026 年开源模型；需要 Gemini 2.5 Pro 或更激进池化。

三种扩展路径出现了。

## 概念

### 路径 1：暴力上下文（Gemini 1.5、Claude Opus）

用硬件砸问题。将上下文扩展到数百万 token，一次前向处理所有内容。

Gemini 1.5 Pro 以 100 万 token 发布；Gemini 1.5 Ultra 到 1000 万；Gemini 2.5 Pro 在 2026 年可靠地处理数小时的视频。论文（arXiv:2403.05530）记录了高达约 950 万 token 的草堆找针回忆率 99.7%。

工程：自定义注意力实现，带内存层级（局部 + 全局 + 稀疏）加上 MoE 专家路由用于长上下文效率。未公开完整细节。非开源。

### 路径 2：环形注意力（LWM、LongVILA）

环形注意力分布式地将长序列分布在设备上，形成一个"环"，每个设备持有一个块。跨整个序列的注意力通过每个设备将其块以环形模式发送给下一个，计算部分注意力并聚合来实现。

LWM（Liu 等人，2024）以这种方式训练了 100 万 token 上下文模型。训练算力随上下文长度线性而非二次地扩展——注意力上的二次惩罚在环的各设备间平摊。

LongVILA（arXiv:2408.10188）将该模式适配到 VLM。1400 帧视频在每帧 192 token = 268k 上下文，通过 8 路并行环形注意力训练。

### 路径 3：Token 压缩（Video-XL、LongVA）

比暴力上下文更便宜：在 LLM 看到序列之前激进压缩。

Video-XL（arXiv:2409.14485）使用视觉摘要 token：每个 N 帧片段产生一个单一的"摘要"token，关注这 N 帧。推理时，LLM 看到每片段一个摘要 token，大幅缩小上下文。

LongVA 通过"长上下文迁移"技术将 LLM 上下文从 200k 扩展到 200 万。在长上下文文本上训练，通过共享表示迁移到长上下文视频。

Token 压缩以特定时间戳的回忆换取可扩展性。模型大体上知道发生了什么，但有时错过精确帧。

### 路径 4：代理式检索（VideoAgent）

不要把完整视频喂给 LLM。相反，将视频视为数据库，用 LLM 查询它。

VideoAgent（arXiv:2403.10517）：

1. LLM 读取问题。
2. LLM 询问检索工具获取相关片段（"给我展示有猫的片段"）。
3. 工具返回匹配片段的时间戳。
4. LLM 通过 VLM 读取这些片段。
5. LLM 合成答案或询问后续查询。

这是应用于长视频的 LLM 作为代理的模式。更便宜的推理（仅编码相关片段），更难的工程（检索质量成为瓶颈）。

### 草堆找针基准

标准长上下文测试：在视频中随机点插入一个独特的视觉或文本标记，然后询问一个需要回忆它的问题。

指标：跨视频时长和标记位置的 Recall@k。

Gemini 2.5 Pro 在最长 90 分钟视频上得分 >99% 回忆率。开源 72B 模型（Qwen2.5-VL-72B、InternVL3-78B）在 30 分钟时得分约 85-90%，超过 60 分钟后下降。

VideoAgent 在 2+ 小时内容上可以匹配或击败原始上下文模型，因为如果工具好检索就能命中标记。

### 选择哪条路径

对于 15 分钟片段、前沿准确率：开源 72B + 原生上下文通常能工作。选择 Qwen2.5-VL-72B。

对于 30 分钟到 1 小时内容：开源选 LongVILA 或 Video-XL；闭源选 Gemini 2.5 Pro。质量门槛很重要——前沿走闭源。

对于 2+ 小时内容：VideoAgent 或类似检索模式。或者，摘要为更小块并喂入分层摘要。

### 2026 年生产模式

在实践中，生产长视频管道是混合的：

1. 在整个视频上运行动态 FPS 采样 + 激进池化（获得 100k token 全局表示）。
2. 传递给 72B VLM 做全局摘要。
3. 如果用户问详细问题，用摘要作为索引运行代理式检索。

这结合了暴力上下文用于全局理解和检索用于局部细节。

## 使用它

`code/main.py`：

- 计算 1 分钟到 3 小时视频在不同 FPS + 池化下的 token 预算。
- 模拟草堆找针运行：在随机时间戳注入标记，提问，评分回忆率。
- 包含一个模拟代理式检索路由器，选择特定片段喂给下游 VLM。

运行预算表，感受规模差距。

## 练习

1. 45 分钟讲座在 1 FPS、每帧 81 token 下。总 token 数？适合哪些模型的上下文？

2. 设计一个草堆找针测试：在第几分钟注入标记，确切的查询格式是什么？

3. 在 1 小时视频上比较暴力上下文 Qwen2.5-VL-72B（80k 上下文）和 VideoAgent（Claude 3.5 + 检索）。哪个在回忆率上赢？哪个在延迟上赢？

4. 环形注意力的内存成本随序列长度和设备数量线性扩展。解释为什么以及如果去掉环形旋转阶段会出什么事。

5. 阅读 Gemini 1.5 第 5 节关于草堆找针。论文在 100 万 vs 1000 万 token 边界上发现了什么关于回忆率的结论？

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 暴力上下文 | "加更多 token 就行" | 将 LLM 上下文扩展到数百万 token；一次处理所有内容 |
| 环形注意力 | "LWM 风格并行" | 分布式注意力模式，每个设备持有一个块并旋转 |
| Token 压缩 | "摘要 token" | 在 LLM 之前通过学习的压缩器减少每片段 token |
| 草堆找针 | "NIH 测试" | 在随机点插入唯一标记，测试时让模型回忆它 |
| 代理式检索 | "LLM 作为查询规划器" | LLM 向检索工具询问相关片段，通过 VLM 读取，合成答案 |
| VideoAgent | "视频的检索模式" | 经典代理式检索设计：问题 -> 工具 -> 片段 -> 答案 |
